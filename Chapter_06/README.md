# Vol.6: 权限堡垒 (Permissions)

## 1. 实战背景 (The Pain Point)

你刚写好一个自动化部署脚本 `deploy.sh`，兴冲冲地输入 `./deploy.sh` 准备上线，结果终端冷冷地回了一句：
`bash: ./deploy.sh: Permission denied`

或者，当你试图连接服务器时，SSH 客户端报错：
`WARNING: UNPROTECTED PRIVATE KEY FILE!`

这些都是 Linux 权限体系在“保护”你。Linux 是一个多用户系统，每个文件都有严格的“身份”和“权限”控制。不懂权限，你在 Linux 世界将寸步难行。

## 2. 核心命令详解 (The Toolkit)

| 命令 | 常用参数 | 说明 |
| :--- | :--- | :--- |
| `ls` | `-l` | 查看详细权限（**必用**） |
| `chmod` | `+x`, `755`, `600` | 修改文件权限 (Change Mode) |
| `chown` | `user:group` | 修改文件所有者 (Change Owner) |
| `sudo` | | 以超级用户身份执行命令 |
| `id` | | 查看当前用户身份和所属组 |

> **查阅手册**: 输入 `man chmod` 查看权限位的详细定义。

## 3. 正则与逻辑 (The Logic)

### 3.1 权限解读 (`drwxr-xr-x`)

在 `ls -l` 的输出中，第一列由 10 个字符组成：

```text
d rwx r-x r-x
^ ^^^ ^^^ ^^^
|  |   |   |
|  |   |   +-- Others (其他人): 可读、可执行
|  |   +------ Group (同组人): 可读、可执行
|  +---------- User (所有者): 可读、可写、可执行
+------------- 类型: d=目录, -=文件
```

### 3.2 数字法 (4-2-1)

权限可以用数字表示，方便加减：
*   **r (Read)** = 4
*   **w (Write)** = 2
*   **x (Execute)** = 1

常见组合：
*   **7 (rwx)** = 4+2+1 (完全控制)
*   **6 (rw-)** = 4+2 (读写，不可执行)
*   **5 (r-x)** = 4+1 (读与执行，不可修改)
*   **4 (r--)** = 4 (只读)

**典型场景：**
*   **755**: 所有者拥有全部权限，其他人只读执行（脚本、程序常见）。
*   **644**: 所有者读写，其他人只读（配置文件常见）。
*   **600**: 只有所有者能读写（SSH 私钥、数据库密码文件）。

## 4. 实验手册 (Hands-on Lab)

请先运行初始化脚本：
```bash
cd Chapter_06/lab

```

进入 `Chapter_06/lab` 目录，完成以下任务：

### 任务 1: 让脚本跑起来
目录下有一个 `deploy.sh` 脚本，尝试运行它 `./deploy.sh`，你会发现报错。
**目标**: 修改权限，使其可以被执行。

### 任务 2: 保护私钥
目录下有一个模拟的 SSH 私钥文件 `id_rsa`。目前的权限太开放了（644），SSH 会拒绝使用它。
**目标**: 将权限设置为 **600**（仅所有者可读写）。

### 任务 3: 开放 Web 目录
目录下有一个 `public_html` 目录，目前的权限是 `700`（只有你能访问）。
**目标**: 将其权限修改为 **755**，以便 Web 服务器（模拟为其他用户）可以读取其中的文件。

## 6. 避坑与原理 (Deep Dive)

*   **执行权限 (`x`)**: 对于文件，表示可以作为程序运行；对于目录，表示**可以进入 (cd)** 该目录。如果你把目录的 `x` 去掉，你就进不去了！
*   **Root 权限**: `sudo` 可以让你临时获得上帝权限。但在生产环境中，滥用 `sudo` 是危险的。尽量使用最小权限原则。
*   **Chown**: `chown` 通常需要 `sudo` 权限才能将文件送给别人。
